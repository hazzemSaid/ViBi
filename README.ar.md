# ViBi (فايبي)

تطبيق تواصل اجتماعي للأسئلة والأجوبة مبني باستخدام Flutter، يدعم خاصية الأسئلة المجهولة.

**اللغات / Languages:**
- 🇬🇧 [English](README.md)
- 🇸🇦 [العربية](README.ar.md)

## 📦 الميزات

### تم إنجازها ✅
- ✅ مصادقة المستخدمين (البريد الإلكتروني/كلمة المرور + تسجيل الدخول بحساب Google)
- ✅ ملفات شخصية للمستخدمين مع صور رمزية وروابط لحسابات التواصل الاجتماعي
- ✅ القصص / الحالات (محتوى يختفي بعد 24 ساعة على غرار إنستغرام)
- ✅ صندوق الأسئلة (استقبال أسئلة مجهولة المصدر)
- ✅ نظام للمنشورات (Feed) والتفاعل معها
- ✅ نظام المتابعة (متابعة / إلغاء متابعة)
- ✅ ميزة البحث (بحث عن المستخدمين والمحتوى)
- ✅ الإشعارات مع دعم الإشعارات الفورية (Push Notifications)

### قيد التطوير 🚧
- 🚧 الرسائل المباشرة (الدردشة)
- 🚧 أدوات إدارة المحتوى والمراقبة
- 🚧 خوارزمية المحتوى الرائج
- 🚧 الدعم للعمل دون اتصال بالإنترنت (Offline Mode)

## 🏗️ البنية البرمجية

يتبع ViBi مبادئ **البنية النظيفة (Clean Architecture)**:

- **طبقة العرض (Presentation)**: واجهة المستخدم وإدارة الحالة باستخدام (Riverpod)
- **طبقة المجال (Domain)**: منطق العمل الأساسي والكيانات  
- **طبقة البيانات (Data)**: مصادر البيانات (Supabase) والمستودعات (Repositories)

### التقنيات المستخدمة
- **الواجهة الأمامية**: Flutter + Dart
- **إدارة الحالة**: Riverpod
- **الخلفية (Backend)**: Supabase (PostgreSQL)
- **المصادقة**: Supabase Auth
- **التوجيه (Routing)**: GoRouter
- **واجهة التطبيق البرمجية**: GraphQL

رغم أننا نرحّب بالمساهمات، إلا أننا نعطي الأولوية للمشاكل التقنية (Issues) وطلبات الدمج (Pull Requests) ذات الجودة العالية. التزامك بالإرشادات التالية يضمن مراجعة أسرع لمساهمتك.

### القواعد الأساسية

- قد لا نقوم بالرد على المشكلة أو طلب الدمج الخاص بك.
- قد نقوم بإغلاق المشكلة أو طلب الدمج دون تقديم توضيحات مفصلة.
- قد نقوم بإغلاق باب النقاشات أو المساهمات مؤقتاً إذا واجهنا ضغطاً كبيراً أو إغراقاً (DDOSed).
- لن نقوم بتقديم الدعم الفني المتعلق بمشاكل بناء وتشغيل التطبيق محلياً (Build issues).

### الإرشادات

- يرجى التحقق من [المشاكل التقنية المفتوحة مسبقاً](https://github.com/yourusername/vibi/issues) قبل فتح مشكلة جديدة.
- قم بفتح مشكلة جديدة (Issue) واترك بعض الوقت لمناقشتها قبل إرسال طلب دمج (PR).
- تجنّب طلبات الدمج (PRs) التي تهدف إلى:
  - تغيير مصطلحات عامة مثل تغيير كلمة "أسئلة" إلى مصطلح آخر في كامل الكود.
  - إعادة هيكلة شاملة للكود، كاستبدال Riverpod بـ Provider مثلاً.
  - إضافة ميزات جديدة كلياً دون مناقشتها مسبقاً مع فريق العمل.

تذكّر دائماً، نحن نخدم مجتمعاً كبيراً من المستخدمين، وعملنا اليومي يتمحور حول سؤال: "ما هي الأولوية القصوى الآن؟". إذا قدمت طلبات دمج مكتوبة بعناية وتحل مشاكل فعلية بشكل مباشر وموجز، فهذه مساهمة رائعة نقدرها. أما عدا ذلك، وبقدر رغبتنا في قبول جميع الأفكار، لا نملك القدرة أو الوقت الكافي لاستيعاب كل شيء. ولهذا السبب بالتحديد وُجدت خاصية تفريع المشروع (Forking)!

### إعداد بيئة التطوير

```bash
# استنساخ المستودع وإعداده
git clone https://github.com/yourusername/vibi.git
cd vibi

# تثبيت الحزم والمكتبات
flutter pub get

# تشغيل التطبيق
flutter run

# تنسيق الكود البرمجي
flutter format lib/ test/

# تشغيل الاختبارات
flutter test
```

### معايير كتابة الكود

- اتبع [دليل كتابة كود Dart](https://dart.dev/guides/language/effective-dart/style) المعتمد.
- اعتمد الرسائل التقليدية والواضحة عند إضافة التعديلات (Commits): `feat:`, `fix:`, `docs:`, `test:`, `refactor:`
- الأمان من القيم الفارغة (Null safety) أمر إلزامي.
- استخدم Riverpod حصرياً لإدارة حالة التطبيق.
- أضف اختبارات برمجية (Tests) للميزات الجديدة التي تقوم ببنائها.
- قم بتحديث التوثيق عند إجراء أي تعديلات على واجهات التطبيق البرمجية (APIs).

### هل أنت مطور مهتم ببناء تطبيقات باستخدام Supabase و Flutter؟

ViBi هو تطبيق تواصل اجتماعي للأسئلة والأجوبة، مفتوح المصدر، مبني على بيئة Supabase و Flutter. من خلال Supabase، ستحصل على قاعدة بيانات قوية وقابلة للتوسع (PostgreSQL)، ونظام مصادقة متكامل، ومزامنة بيانات بالوقت الفعلي، والكثير—مما يمنحك القدرة على بناء تطبيقات مذهلة دون التقيد بالمنصات المغلقة أو الاحتكارية.
